## Before you start

- [ ] kubectl **admin** access to the cluster (read-only can't do the migration steps)
- [ ] ArgoCD UI + admin creds — `https://argocd.<domain>` (creds from dash)
- [ ] Gitea UI + admin creds — `https://gitea.<domain>` (creds from dash)
- [ ] Cluster is on a v2.x cluster-forge release, kgateway + minio still live


> Replace `<domain>` with the cluster domain everywhere below.

## Part 1 — kgateway → envoy-gateway

### 1. Run the gateway migration job
While kgateway is **still live**. Copies the TLS secret and labels the first node.

```bash
kubectl apply -f https://raw.githubusercontent.com/silogen/cluster-forge/v2.2.2/scripts/utils/envoy-gateway-migration.yaml

# Phase 1 — init container copies the TLS secret (watch this first):
kubectl logs -f job/envoy-gateway-migration -n envoy-gateway-system -c tls-copy

# Phase 2 — main container labels the node (watch after Phase 1 finishes):
kubectl logs -f job/envoy-gateway-migration -n envoy-gateway-system -c node-labeling
```
✅ Wait for **`MIGRATION COMPLETE`** in the Phase 2 log before continuing.

> The job runs in two phases. Right after apply the pod is `PodInitializing` and the default
> `kubectl logs` (which targets `node-labeling`) errors with *"container is waiting to start:
> PodInitializing"* — that's normal. Tail `-c tls-copy` first, then `-c node-labeling`.

### 2. Edit `values.yaml` in Gitea
Open Gitea → `cluster-org/cluster-values` → `values.yaml` → Edit. Make all changes, then
**commit + push**. (Auto-sync stays on — that's fine.)

**a) Bump the version:**
```yaml
clusterForge:
  targetRevision: v2.2.2
```

**b) Swap the app list** — under `enabledApps`, remove:
```
gateway-api, kgateway, kgateway-config, kgateway-crds,
minio-operator, minio-tenant, minio-tenant-config
```
and add:
```
ai-gateway-discovery, inference-extension-crds,
envoy-gateway, envoy-gateway-config, envoy-ai-gateway-crds, envoy-ai-gateway,
seaweedfs-crds, seaweedfs-operator, seaweedfs-config
```

**c) GPU family — depends on the cluster:**
- **Radeon:** add the radeon model source under `apps:`
  ```yaml
  apps:
    aim-cluster-model-source:
      valuesObject:
        hardwareFamilies:
          - radeon
  ```
- **Instinct (MI300X, e.g. staging):** **skip this** — Instinct is the chart default.

### 3. Change **`cluster-forge`** application source revision

1. Open **`https://argocd.<domain>`** and log in (user `admin`, password from dash).
2. Click the **`cluster-forge`** application to open it.
3. Click **Details** (top toolbar) → open the **Manifest** tab → click **EDIT**.
4. Under `spec.sources`, find the source with **`path: root`** and change its
   `targetRevision` to **`v2.2.2`**:
   ```yaml
   spec:
     sources:
       - repoURL: http://gitea-http.cf-gitea.svc:3000/cluster-org/cluster-forge.git
         path: root
         targetRevision: v2.2.2      # <-- change this line
         helm:
           valueFiles:
             - values.yaml
             - values_medium.yaml       # use the right size for your cluster
             - $cluster-values/values.yaml
       - ref: cluster-values            # leave this source unchanged
         repoURL: http://gitea-http.cf-gitea.svc:3000/cluster-org/cluster-values.git
         targetRevision: main
   ```
   Leave the **`cluster-values`** source untouched. Click **SAVE**.
5. Back on the app, click **REFRESH** → choose **Hard Refresh** from the dropdown.
6. Auto-sync then rolls it out (if it doesn't start on its own, click **SYNC** → **SYNCHRONIZE**).

### 4. Watch the apps roll out
In ArgoCD, watch the new apps come up (envoy-*, seaweedfs-*, ai-gateway-discovery). The old
kgateway/minio stay running for now — that's expected, they are **not** auto-deleted.

### 5. Free the LoadBalancer IP
Once envoy pods are Running but its `https` Gateway shows `Programmed=False` (kgateway still
owns the IP), delete **only the kgateway Gateway object**. This releases the LB IP so envoy
can take it, with minimal blast radius. Leave the kgateway namespace and CRDs in place — they
come out in **Cleanup** at the end.

```bash
kubectl delete gateway https -n kgateway-system
```

Deleting the Gateway garbage-collects its LoadBalancer Service (owner reference), which frees
the IP. Within a few seconds envoy's `https` Gateway picks it up and goes `Programmed=True`:
```bash
kubectl get gateway -A    # envoy-gateway-system/https now Programmed=True at the LB IP
```

### 6. Re-point app routes still bound to kgateway
After the IP handoff, some apps' HTTPRoutes may still parent the deleted `kgateway-system/https`
gateway — they `404` until re-pointed at envoy. Find them:

```bash
kubectl get httproute -A -o jsonpath='{range .items[*]}{.metadata.namespace}/{.metadata.name} {range .spec.parentRefs[*]}{.namespace}/{.name} {end}{"\n"}{end}' | grep kgateway-system
```
(The `minio` route can be ignored — minio is removed in Part 2.)

**airm** — its route's gateway namespace comes from a helmParameter that defaults to kgateway.
Add it under `apps.airm` in cluster-values `values.yaml` (keep any existing params):
```yaml
apps:
  airm:
    helmParameters:
      - name: airm-api.kgateway.namespace
        value: envoy-gateway-system
```
Commit + push, then in ArgoCD **sync `cluster-forge` (umbrella) first, then `airm`** — the
param only reaches airm's route after the umbrella re-renders airm's Application spec.

**aiwb** — no param needed; the chart already targets envoy. Just **sync the `aiwb` app**.

Verify the routes moved and the APIs respond (test a real path, not `/` — APIs `404` on root):
```bash
kubectl get httproute -A -o jsonpath='{range .items[*]}{.metadata.name} {range .spec.parentRefs[*]}{.namespace}/{.name} {end}{"\n"}{end}' | grep -v kgateway-system
curl -sk -o /dev/null -w "%{http_code}\n" --resolve airmui.<domain>:443:<LB-IP>  https://airmui.<domain>         # 307
curl -sk -o /dev/null -w "%{http_code}\n" --resolve aiwbapi.<domain>:443:<LB-IP> https://aiwbapi.<domain>/docs  # 200
```

> Known issue: `aiwb-ui-route` ships with `hostnames: null`, so the aiwb **UI** still `404`s on
> its hostname even after re-pointing. This is an upstream chart bug (identical on the reference
> cluster), not something this upgrade introduces — flag it, don't block on it.

### 7. Check Part 1 is healthy
```bash
kubectl get pods -n envoy-gateway-system     # envoy pods Running
kubectl get gateway -A                        # https Gateway Programmed=True at the LB IP
kubectl get aimclustermodelsource -A          # model source exists
```
- [ ] envoy pods Running, `cluster-tls` secret in `envoy-gateway-system`
- [ ] envoy `https` Gateway **Programmed=True** at the old LB IP
- [ ] Gitea + ArgoCD reachable over HTTPS
- [ ] All ArgoCD apps Synced/Healthy
- [ ] **minio still running** (needed for Part 2 — do not delete yet)

### 8. Generate the deploy profiles
The deploy UI is driven by `AIMClusterProfile` objects, which aim-engine produces by running
discovery against each `AIMClusterModelSource`. On an in-place upgrade the existing sources
**don't re-discover by themselves**, so you re-trigger discovery by **deleting them all and
letting ArgoCD recreate them fresh** — that's the whole step.

```bash
# 1. delete every source
kubectl delete aimclustermodelsource --all

# 2. recreate from git: the aim-cluster-model-source app has auto-sync OFF, so SYNC it
#    (ArgoCD UI: open the aim-cluster-model-source app → SYNC → SYNCHRONIZE)

# 3. confirm they came back and discovery produced profiles (~1-2 min)
kubectl get aimclustermodelsource          # sources back, STATUS=Ready, MODELS > 0
kubectl get aimclusterprofiles -A          # profiles appear, STATUS=Ready
```
✅ Done when the sources are recreated `Ready`, `aimclusterprofiles` are `Ready` with `HARDWARE`
matching your GPU (e.g. `1 x R9700`), and the deploy UI lists models without a "not supported on
your hardware" banner.

> The `aim-cluster-model-source` app does **not** auto-sync, so the sources will **not** come
> back on their own after the delete — you must sync the app in step 2.

> Ignore the per-`AIMClusterServiceTemplate` `Discovered=False` condition — the **profile
> count** is the signal that matters.

**Do not start Part 2 until Part 1 is fully healthy.**

## Part 2 — minio → seaweedfs

### 1. Mirror all buckets
The job auto-discovers every bucket in MinIO and mirrors each to a same-named bucket in
SeaweedFS — one apply does the whole migration:
```bash
kubectl apply -f https://raw.githubusercontent.com/silogen/cluster-forge/v2.2.2/scripts/utils/mirror-minio-to-seaweedfs-job.yaml
kubectl logs -f job/minio-to-seaweedfs-mirror-job -n default
```
✅ Wait for **`Mirror operation completed successfully!`** (job self-deletes 5 min later).

> To mirror just one bucket instead, add a `SOURCE_BUCKET` env var to the Job before applying.

### 2. Validate the copy
The log lists each bucket and its target contents. Spot-check object counts per bucket:

### Fetch credentials
```bash
S3_API_KEY=$(kubectl get secret default-user -n "$MINIO_NAMESPACE" -o jsonpath="{.data.API_ACCESS_KEY}" | base64 -d)
echo $S3_API_KEY
S3_SECRET_KEY=$(kubectl get secret default-user -n "$MINIO_NAMESPACE" -o jsonpath="{.data.API_SECRET_KEY}" | base64 -d)
echo $S3_SECRET_KEY
```
#### Log in to the minio default pool pod
```bash
kubectl exec -n minio default-minio-pool-0-0 -it -- bash
``` 

```bash
$ mc alias set myminio http://localhost:9000 "<paste S3_API_KEY>" "<paste S3_SECRET_KEY>"

>> Added myminio successfully.

$ mc ls myminio/default-bucket --recursive | wc -l

>> some number of objects

$ mc alias set myseaweed http://filer-s3.seaweedfs-instance.svc.cluster.local "<paste S3_API_KEY>" "<paste S3_SECRET_KEY>"

Added myseaweed successfully.

$ mc ls myseaweed/default-bucket --recursive | wc -l

>> some number of objects, same value as above
``` 

Compare against MinIO — they should match.

### 3. Decommission minio
Only **after** seaweed data is confirmed. minio was already removed from `enabledApps` in
Part 1, so let ArgoCD prune it (or delete the minio namespaces manually). Until then minio
stays as a fallback.

## Cleanup — remove leftover kgateway

After Part 1 is healthy (envoy `https` Programmed=True on the LB IP) and the cluster has been
stable, remove the now-unused kgateway namespace and CRDs. There are no kgateway ArgoCD apps
left to recreate them, so this is safe.

```bash
kubectl delete namespace kgateway-system

kubectl delete crd \
  gateways.gateway.kgateway.dev gatewayextensions.gateway.kgateway.dev \
  trafficpolicies.gateway.kgateway.dev backendconfigpolicies.gateway.kgateway.dev \
  backends.gateway.kgateway.dev directresponses.gateway.kgateway.dev \
  gatewayparameters.gateway.kgateway.dev httplistenerpolicies.gateway.kgateway.dev \
  --ignore-not-found
```

> minio decommission is handled separately in **Part 2, step 3** — only after SeaweedFS data
> is confirmed.

## If something goes wrong (rollback)

- **Part 1:** revert `clusterForge.targetRevision` and the `enabledApps` change in
  `values.yaml`, revert the umbrella source patch. kgateway comes back; delete
  `envoy-gateway-system` namespace if needed.
- **Part 2:** the mirror job only **copies** — it never deletes minio. So you can keep using
  minio and just re-run the mirror.
